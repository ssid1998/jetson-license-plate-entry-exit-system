\chapter{From Development to Deployment}
\label{sec:from_dev_to_dep}
\index{Deployment}

\noindent
This section documents the pragmatic step from a trained license plate pipeline to a runnable application on the NVIDIA Jetson Nano.
The focus is not DevOps theory, but the concrete runtime structure: \textbf{how inference results are represented}, which \textbf{tools} are required on-device, and how the system \textbf{saves and loads state} so that it can be restarted reliably.

\section{Data Structure}
\label{sec:devtodep_data_structure}
\index{Deployment!Data Structure}

\noindent
After inference, the application does not operate on raw tensors; it operates on small Python objects that are easy to display, validate, and persist.
The deployed Jetson dashboard (\FILE{Code/LicencePlateDetection/jetson-inference/python/www/flask/dashboard2.py}) uses plain Python types (\PYTHON{dict}, \PYTHON{list}, \PYTHON{tuple}) and two deployment-relevant runtime outputs: a persistent SQLite database of plate visits (billing state) and the most recent encoded video frame for the web stream.

\subsection{Runtime objects and their roles}
\index{Deployment!Data Structure!Runtime objects}

\noindent
At runtime, the Jetson application is organized into the following objects/modules:
\begin{itemize}
	\item \textbf{Camera source:} frames are captured via OpenCV (\PYTHON{cv2.VideoCapture}), trying camera index 0 and falling back to 1, and configured to \(640\times480\) for the live stream.
	\item \textbf{TensorRT inference:} the detector is deployed as a TensorRT engine (\PYTHON{best.engine}) and executed through TensorRT + PyCUDA (manual binding buffers in \FILE{Code/LicencePlateDetection/jetson-inference/python/www/flask/dashboard2.py}).
	\item \textbf{OCR reader:} \PYTHON{easyocr.Reader} runs in CPU mode and outputs candidate text strings; the final plate string is produced using an allowlist and German-plate formatting/validation logic (prefix corrections, allowed city codes, and a numeric suffix constraint).
	\item \textbf{Persistent session state (SQLite \PYTHON{visits} table):} plate visits are stored in \PYTHON{parking.db} with the schema \PYTHON{(plate, entry\_ts, last\_seen\_ts)}; inserts/updates are written during inference and read back by the billing API.
	\item \textbf{Stream buffer (\PYTHON{output\_frame}):} the latest camera frame with overlays encoded as JPEG bytes for MJPEG streaming over HTTP.
	\item \textbf{Web API layer:} Flask serves the dashboard HTML and exposes JSON billing data (\PYTHON{/api/billing}), a video stream endpoint (\PYTHON{/video\_feed}), and an operational shutdown action (\PYTHON{/api/shutdown}).
\end{itemize}

\subsection{Billing event representation (post-inference)}
\index{Deployment!Data Structure!Billing event}

\noindent
The core unit that flows through the deployed system is a \textbf{billing session record} for a recognized plate string.
It is represented by one row in the SQLite \PYTHON{visits} table (stored at \PYTHON{DB\_PATH}):

\section{Tools}
\label{sec:devtodep_tools}
\index{Deployment!Tools}

\noindent
The deployed application depends on a small number of tools that must be available on the Jetson Nano for a reproducible run:
\begin{itemize}
	\item \textbf{Python runtime + pip:} runs the application and installs dependencies.\\
	\noindent Deployment dependencies for the Flask dashboard: \PATH{Code/LicencePlateDetection/jetson-inference/python/www/flask/requirements.txt}.
	
	\item \textbf{JetPack / L4T + CUDA stack:} provides the GPU runtime required by Jetson inference libraries.
	
	\item \textbf{TensorRT:} runs the optimized detector as a serialized engine file (\PYTHON{best.engine}).\\
	\noindent Loaded in: \PATH{Code/LicencePlateDetection/jetson-inference/python/www/flask/dashboard2.py}.
	
	\item \textbf{PyCUDA:} allocates buffers and executes the TensorRT context from Python.\\
	\noindent Usage reference: \PATH{Code/LicencePlateDetection/jetson-inference/python/www/flask/dashboard2.py}.
	
	\item \textbf{OpenCV:} image preprocessing (resize, color conversion), overlay rendering, and JPEG encoding for the web stream.
	
	\item \textbf{EasyOCR (CPU mode):} reads text from plate crops using an allowlist to reduce false characters.
	
	\item \textbf{Flask:} serves the local web UI and provides API endpoints for billing state.\\
	\noindent Entrypoint: \PATH{Code/LicencePlateDetection/jetson-inference/python/www/flask/dashboard2.py}.
	
	\item \textbf{SQLite (Python stdlib):} persists billing state to a file-backed database (\PYTHON{parking.db}) at \PYTHON{DB\_PATH}.
	
	\item \textbf{Sudo / power management:} optional operational dependency if the dashboard shutdown action is used.\\
	\noindent Endpoint: \PYTHON{/api/shutdown} calls \texttt{sudo poweroff}.
	
	\item \textbf{Git:} used to synchronize the application code and model artefacts between development and the Jetson Nano.
\end{itemize}

\section{Saving (Persistence)}
\label{sec:devtodep_saving}
\index{Deployment!Saving}

\noindent
For a deployed edge system, persistence means:
\textbf{results survive application restarts}.

\noindent
In the currently deployed Jetson dashboard, persistence is implemented using a
file-backed SQLite database, so billing state survives restarts.

\noindent
Implementation reference:
{\raggedright
	\PATH{Code/LicencePlateDetection/jetson-inference/python/www/flask/dashboard2.py}\par}


\subsection{Persisted artefacts (files)}
\index{Deployment!Saving!Persisted artefacts}

\noindent
The following artefacts exist as files and survive restarts:
\begin{itemize}
	\item \textbf{TensorRT engine file:} stored on SSD at \PATH{/media/myssd/jetson-inference/data/networks/best.engine} and loaded at startup (\PYTHON{ENGINE\_PATH}).
	\item \textbf{Billing database:} stored on SSD at \PATH{/media/myssd/jetson-inference/python/www/flask/parking.db} and updated at runtime (\PYTHON{DB\_PATH}).
	\item \textbf{Application code:} stored in the Git repository and executed directly on the Jetson Nano.
\end{itemize}

\subsection{Non-persistent state (current implementation)}
\index{Deployment!Saving!Non-persistent state}

\noindent
The following runtime state is volatile and will be lost after stopping/restarting the process:
\begin{itemize}
	\item \textbf{Latest stream buffer:} \PYTHON{output\_frame} (MJPEG output).
	\item \textbf{Recent overlay state:} last accepted plate text and timestamp used for on-screen feedback (e.g., \PYTHON{last\_valid\_plate}).
\end{itemize}

\noindent
This means that after a restart, the dashboard resumes with the existing database state, while the live stream and transient UI overlays restart from a clean state.

\section{Loading (Startup and Restart)}
\label{sec:devtodep_loading}
\index{Deployment!Loading}

\noindent
At startup (or after a crash), the deployed system reconstructs the minimum required state in a deterministic order:

\subsection{Model and OCR loading}
\index{Deployment!Loading!Models}

\noindent
At startup, the TensorRT engine is loaded from \PYTHON{ENGINE\_PATH} and a TensorRT execution context is created.
In parallel, the EasyOCR reader is initialized once (CPU mode) and reused for all subsequent plate crops (\PYTHON{reader = easyocr.Reader(..., gpu=False)} in \FILE{Code/LicencePlateDetection/jetson-inference/python/www/flask/dashboard2.py}).

\index{Deployment!Loading!Configuration}

\noindent
In the current student-project implementation, runtime parameters are kept as explicit Python variables
at the top of the scripts. Examples include \PYTHON{ENGINE\_PATH}, \PYTHON{DB\_PATH},
\PYTHON{CONFIDENCE\_THRESHOLD}, \PYTHON{OCR\_COOLDOWN}, and the operational
\PYTHON{SYSTEM\_PASSWORD}.

\noindent
These parameters are defined in:\par
{\raggedright
	\PATH{Code/LicencePlateDetection/jetson-inference/python/www/flask/dashboard2.py}\par}

\noindent
This approach keeps the deployed system transparent: a restart reproduces the same behaviour sas long as the code version is unchanged.




\subsection{Loading historical information (current state)}
\index{Deployment!Loading!History}

\noindent
Historical information is loaded from SQLite at startup: the database file is opened (and created if missing), the \PYTHON{visits} table is ensured to exist, and existing rows are available immediately to the billing API (\PYTHON{/api/billing}).

\subsection{Operational restart procedure (Jetson Nano)}
\index{Deployment!Loading!Restart procedure}

\noindent
For the project demo setup, restarting is performed by stopping the application and launching it again:
\begin{itemize}
	\item \textbf{Flask dashboard:} start the entrypoint shown below; stop with \SHELL{Ctrl+C} in the terminal.
	
	\noindent\SHELL{cd Code/LicencePlateDetection/jetson-inference/python/www/flask \;\&\&\; python3 dashboard2.py}
	
	\item \textbf{Access from a browser:} open \PATH{http://<jetson-ip>:8055/} to view the live stream and billing table.
\end{itemize}


\noindent
An auto start/stop mechanism (e.g., a \texttt{systemd} service) is not configured yet and remains future work for the project.
