\chapter{Domain System - Software} \label{ch:domain_system_software}
\section{Application Domain Knowledge: Software}
\label{sec:software_domain}

While hardware provides the physical substrate for computation tasks, software domains serve as the cognitive logic of the License Plate Recognition (LPR) system. In embedded edge AI, software architectures are not only defined by application code but also by rigid constraints of underlying Operating System (OS).

This section explains the software ecosystem while establishing the Operating System limitations along with defining the engineered deployment pipelines, the memory management architecture, and the specific application logic used by the YOLO object detector and OCR models.

% ---------------------------------------------------------------------------
% SUBSECTION: DESCRIPTION & OS
% ---------------------------------------------------------------------------
\subsection{System Description: The Legacy Ecosystem}

Primarily, a software constraint in the context of this project is the NVIDIA JetPack SDK version supported by the Jetson Nano. As opposed to modern desktop environments, the Jetson Nano is locked to **JetPack 4.6**, which is built upon **Ubuntu 18.04 LTS**. This imposes a "Legacy Software Ecosystem" challenge that dictates every subsequent architectural decision.

As visualized in Figure \ref{fig:software_stack}, the dependency layer creates a critical bottleneck. Modern AI libraries, such as \texttt{Ultralytics YOLOv8} and \texttt{Pandas 2.0}, natively require Python 3.8 or newer. The Jetson's default Python 3.6 environment is incompatible with these standard pip packages. Consequently, the "System Description" includes not just the standard libraries, but a set of custom "Monkey Patching" scripts engineered to map deprecated API calls to legacy endpoints, ensuring runtime stability.

\subsection{Software Design and Installation Pipeline}

To overcome the computational and software limitations of the edge device, we adopted a \textbf{Split-Architecture Design}. Training a Convolutional Neural Network (CNN) directly on the Jetson Nano is unfeasible due to memory exhaustion. Therefore, the "Installation" phase is not a single step but a pipeline spanning two distinct computing environments: the Training Host (High-Performance PC) and the Inference Host (Jetson Nano).

\vspace{0.5cm}

% --- VISUAL 2: DEPLOYMENT PIPELINE FLOWCHART ---
\begin{figure}[htbp]

	\caption{The Split-Architecture Deployment Pipeline used to bypass hardware limitations.}

The workflow illustrated in Figure \ref{fig:deployment_pipeline} addresses the "Cross-Compilation" challenge. The YOLOv8 model is trained on a powerful workstation. The resulting weights (\texttt{.pt} file) are exported to the \textbf{ONNX} (Open Neural Network Exchange) format. This representation is transferred to the Jetson Nano, compiled into a \textbf{TensorRT} engine file. This binary format is optimized for the Nano's Maxwell GPU architecture, providing a 40-50\% improvement in inference speed (FPS) as oppposed to running raw PyTorch code.

\subsection{Data Structure and Memory Architecture}

A defining characteristic of the "Software Domain" on embedded devices is the scarcity of Random Access Memory (RAM). The Jetson Nano possesses a unified memory architecture where the CPU and GPU share a single 4GB pool. This creates a "Zero-Sum Game": every megabyte used by the Ubuntu OS is a megabyte denied to the AI model.

During initial testing, the simultaneous loading of the OS ($\approx 1.2$ GB), the YOLOv8 model ($\approx 1.5$ GB), and the EasyOCR engine ($\approx 1.8$ GB) resulted in a total demand of $\approx 4.5$ GB. This exceeded physical capacity, causing the Linux kernel to invoke the \textbf{OOM (Out of Memory) Killer}, terminating the application immediately.

To resolve this, we engineered a virtual memory expansion using a \textbf{6GB Swap Partition} on the external SSD.

\vspace{0.5cm}

% --- VISUAL 3: MEMORY MAP DIAGRAM ---
\begin{figure}[htbp]

As shown in Figure \ref{fig:memory_architecture}, the Swap partition allows the OS to offload inactive memory pages (like the GUI desktop manager) to the SSD, freeing up the fast physical RAM for the active CUDA tensors required by YOLOv8. While SSD access is slower than RAM, this trade-off is acceptable for background processes and enables the system to run without crashing.

\subsubsection{Data Types and Database Structure}
Data handled within the software comprises of structured entry/exit events.In order to promote persistence through power cycles, a local \textbf{SQLite} database is implemented. Schema  normalization is performed aiding in database table simplification optimized for CRUD applications while license plated timestamps are being recorded and processed in the parking application:

\begin{itemize}
	\item \textbf{Table:} \texttt{vehicle\_logs}
	\item \textbf{Columns:} 
	\begin{itemize}
		\item \texttt{id} (INTEGER PRIMARY KEY)
		\item \texttt{license\_plate} (TEXT, Indexed for O(1) retrieval)
		\item \texttt{entry\_timestamp} (DATETIME, ISO 8601 format)
		\item \texttt{exit\_timestamp} (DATETIME, Nullable)
		\item \texttt{fee\_calculated} (FLOAT)
	\end{itemize}
\end{itemize}

This structure allows the billing logic to perform a simple SQL query: \texttt{SELECT entry\_timestamp FROM vehicle\_logs WHERE license\_plate = ? AND exit\_timestamp IS NULL}. If a record is found, the car is exiting; if not, it is entering.

\subsection{Protocols and Remote Development Strategy}

An important milestone was solving challenges with respect to using standard modern development tools, unusable in the "Configuration" domain. The VS Code Remote Server recently updated its requirements to demand \texttt{glibc 2.28}, whereas the Jetson's OS is locked to \texttt{glibc 2.27}. This incompatibility rendered standard remote debugging impossible.

To circumvent this, we architected a protocol switch from the standard VS Code Server agent to a lightweight \textbf{SSH-FS (Secure Shell File System)} implementation.

\vspace{0.5cm}

% --- VISUAL 4: REMOTE PROTOCOL DIAGRAM ---
\begin{figure}[htbp]
	\centering
	\input{../../tikz/3DomainKnowledgeHardware/DomainSystemSoftware/RemoteProtocol.tikz}
	\caption{Remote Development Protocol: Switching from Binary Agents to File System Mounting.}
	\label{fig:remote_protocol}
\end{figure}

As depicted in Figure \ref{fig:remote_protocol}, instead of running a binary agent on the Jetson (which fails due to library versioning), we mount the Jetson's filesystem locally on the development laptop. This allows us to edit code using the laptop's powerful CPU and modern libraries, while the files are physically saved to the Jetson's SSD in real-time. Execution is then triggered via a standard SSH terminal, which requires very few dependencies.

\subsubsection{Conclusion of Software Analysis}
The Software Domain of this project is characterized by adaptation. The application logic (YOLO detection, Billing) is robust, but it sits atop a fragile legacy foundation. The successful operation of the system serves as a proof-of-concept that modern AI workloads can be back-ported to older edge hardware through rigorous memory management (Swap), strategic pipeline design (TensorRT), and protocol flexibility (SSH-FS).
