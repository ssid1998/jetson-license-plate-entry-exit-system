\section{Application Domain Knowledge: Hardware}
\label{sec:hardware_domain}

Development of a robust License Plate Recognition (LPR) system mandates a precise alignment between the application domain knowledge and the hardware implementation. This section describes hardware systems, analyzing essential requirements derived from the physical domain (the road environment) and technical specification alignment of requirements with the chosen embedded platform. Hardware choice stresses on optimal architecture; edge device hardware constraints dictate complexity of the neural networks that can be deployed, in addition to the optical characteristics of the sensors that determine recognition accuracy.

Hardware architecture and specification focuses on a hierarchical system designed to meet real-time image processing constraints. Further subsections explore system description, physical design choices, interface connectivity and operability, data properties, and operational limitations.

\subsection{System Description and Hardware Taxonomy}

To understand the scope of the hardware implementation, we first classify the system components into a hierarchical taxonomy. The "Complete System" is composed of the interaction between the controllable Hardware Domain (the device) and the uncontrollable Environment (the application context). This distinction is vital because software calibration can only compensate for environmental variance up to the physical limits of the hardware sensors.

Figure \ref{fig:hardware_taxonomy} illustrates that the Hardware Domain is based on the \textbf{NVIDIA Jetson Nano}, specifically the reComputer J1010 module. This decision is governed by  parallel computing capabilities required to execute Deep Learning models (specifically YOLOv8) at the edge. Typical micro-controllers (such as Arduino) lack matrix multiplication units (CUDA cores) essential for such tasks. A complete desktop GPU is impractical due to power consumption for license based recognition tasks, especially on road-side deployments.

The acquisition subsystem (Microsoft LifeCam), serving as the sensory interface, generates image frames that functions as the foundational basis for dissecting pixels and transforming pixel information into digital matrices. Conversion quality largely depends on the quality of photo frames and pixel information."Acquisition" node failures in capturing a clear image due to motion blur or poor focus handicaps downstream computing results. Thus, the hardware taxonomy emphasizes the dependency of the software on the physical fidelity of the hardware sensors.

\subsection{Hardware Design and Physical Dimensions}

Physical system design focuses on the form factor and the integration of the entire system into the deployment environment so that it is functional. The \textbf{Seeed Studio reComputer J1010} carrier board is utilized, offering flexibility compared to the standard NVIDIA Developer Kit. This reduction in size is crucial for embedding the system into compact enclosures or mounting it directly onto traffic infrastructure.

Thermal management and port accessibility are important considerations for form factor choices. Figure \ref{fig:j1010_schematic} details the specific layout of the J1010 board. In comparison with the standard developer kit, this board aligns all major I/O ports on a single edge, thus aiding in enclosure design.

The following schematic explains key architectural choices:
\begin{itemize}
	\item \textbf{USB 3.0 Alignment:} All USB 3.0 ports are positioned centrally. This facilitates for high-bandwidth between peripherals (like the LifeCam or additional storage).
	\item \textbf{Thermal Design:} The heatsink is placed centrally (coordinates 3,2.5 to 9,5.5 in the schematic relative unit) where unobstructed vertical airflow is an essential requirement. In order to avoid overheating, cables must not cross this thermal zone. 
	\item \textbf{Power Delivery:} The J1010 uses a USB-C PD input (Far Left) unlike older models with barrel jacks. This allows for the standardization of power supply, although it imposes necessary requirements for high-quality 5V/3A power brick to prevent brownouts during GPU surges.
\end{itemize}

\subsection{Interfaces and System Connectivity}

 "Interfaces" forms the third domain of hardware analysis. A license plate recognition system is fundamentally a pipeline: photons enter via the camera, data flows to the Jetson, processing occurs, and results output to the display. The integrity of this pipeline depends on the bandwidth and stability of the physical interfaces connecting the components.

We map the connectivity into two primary flows: the \textbf{Data Flow} (USB/CSI) and the \textbf{Power Flow} (USB-C).

% --- VISUAL 3: LIFECAM IMAGE ---
\begin{figure}[h!]
	\centering
	\includegraphics[width=0.45\linewidth]{Lifecam/cam1.jpg}
	\caption{Microsoft LifeCam Show — front view of the camera module used in the project \cite{microsoft_lifecam_datasheet}.}
	\label{fig:lifecam_front}
\end{figure}

The visual sensor, the \textbf{Microsoft LifeCam} (shown in Figure \ref{fig:lifecam_front}), acts as the primary data acquisition node. It connects via the USB 2.0 interface. While the J1010 carrier board supports USB 3.0 speeds (up to 5 Gbps), the camera's output (720p at 30fps) is well within the bandwidth limits of USB 2.0 (480 Mbps). A handshake mechanism between host (Jetson Nano) and the camera interface is facilitated by the USB protocol. The negotiation between video formats MJPEG and YUVV is established here. 

USB camera selection as opposed to a MIPI-CSI camera selection was deliberate. CSI cameras promise lower latency through direct connections with the ISP (Image Signal Processor). However,  USB cameras are better in terms of flexibility. A cable length of  1.5-meters allows the camera to mount successfully on an external pole or stand, while the computing unit remains enclosed safely. Furthermore, the Microsoft LifeCam includes a built-in noise-canceling microphone (unused in this visual application) and a glass lens element, which provides superior optical clarity and scratch resistance compared to standard plastic lenses found on cheaper modules.

\subsubsection{Interface Bandwidth Analysis}
To ensure the system does not bottleneck, we calculate the required data throughput.
\begin{equation}
	\text{Bandwidth} = \text{Width} \times \text{Height} \times \text{Channels} \times \text{FPS}
\end{equation}
For a resolution of $1280 \times 720$, with 3 color channels (RGB) at 30 FPS:
\[ 1280 \times 720 \times 3 \text{ bytes} \times 30 \approx 82,944,000 \text{ bytes/sec} \approx 79 \text{ MB/s} \]
Since USB 2.0 has a practical effective throughput of $\approx 35-40$ MB/s, the raw data cannot be transmitted uncompressed. The LifeCam hardware performs onboard MJPEG compression to reduce this bandwidth requirement to $\approx 5-10$ MB/s, allowing reliable transmission even over longer cable runs.

\subsubsection{Analysis of Data Quantity}
The quantity of data dictates the computational load. The system is configured to capture at High Definition (HD) 720p. Such a  resolution offers balance: enough to resolve characters on a license plate from a distance of 3-5 meters, although lower in for the YOLOv8-Nano model to perform inference at real-time speeds (>20 FPS). Higher resolutions, such as 1080p, tend to increase pixel counts by 2.25x, which would lead to frame rates dropping below the threshold needed for successfully capturing moving vehicles.

\subsubsection{Analysis of Data Quality}
Quality refers to the fidelity of the pixels. A critical hardware property identified in Table \ref{tab:hardware_properties} is the **Rolling Shutter**. Unlike a Global Shutter, which exposes all pixels simultaneously, a rolling shutter exposes the image line by line. For fast-moving vehicles (the "Complete System" context), this can induce a skewing artifact known as the "Jello Effect," where the license plate appears slanted or parallelogram-shaped. 

This hardware characteristic imposes a strict constraint on the application domain: the relative speed between the camera and the vehicle must be minimized. While the LifeCam provides excellent color reproduction and low-light sensitivity for a webcam, its rolling shutter mechanism makes it less suitable for high-speed highway monitoring compared to specialized industrial Global Shutter cameras. However, for parking lot entry/exit scenarios (low speed), the distortion is negligible.

\subsubsection{Computational Constraints}
The NVIDIA Jetson Nano possesses 4GB of unified memory. This memory is shared between the CPU and the GPU. This is a "hard constraint." The operating system (Ubuntu with GUI), the graphical interface, and the Python runtime consume approximately 1.5GB to 2GB at idle. This leaves only ~2GB for the YOLOv8 model weights and the input video buffers. If this limit is exceeded, the system forces data into "swap memory" (using the slow SD card as RAM), which destroys real-time performance. Therefore, the hardware directly dictates that we must use optimized model variants (e.g., YOLOv8-Nano or Small) rather than the Large variants.

\subsubsection{Environmental and Physical Constraints}
As shown in the red branch of Figure \ref{fig:hardware_constraints}, lighting is the most volatile variable. The hardware sensor offers limited dynamic range visibility. A white license plate might cause light to reflect in high sunlight levels, leading to "whiteout" or even overexposure. Conversely, the sensor may gain-up during nighttime, introducing ISO noise misleading the OCR algorithm. The solution regarding the problem of "Calibration" then involves setting fixed exposure values or using software-based histogram equalization, in lieu of hardware inabilities to account for such constraints.

\subsubsection{Conclusion}
In light of the above, the hardware application domain knowledge  establishes that although the NVIDIA Jetson Nano and Microsoft LifeCam offer a capable foundation for License Plate Recognition, a successful deployment is dependent on strongly adhering to hardware interface specifications (Power/USB), efficient management of data throughput (Resolution/Compression), and a thorough understanding of physical constraints (Heat/Memory). This hardware analysis sets the foundation for subsequent software development phases.
