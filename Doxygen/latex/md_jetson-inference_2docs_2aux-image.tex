To convert images between data formats and/\+or colorspaces, see the Color Conversion section below.\hypertarget{md_jetson-inference_2docs_2aux-image_autotoc_md60}{}\doxysubsection{\texorpdfstring{Image Allocation}{Image Allocation}}\label{md_jetson-inference_2docs_2aux-image_autotoc_md60}
To allocate empty GPU memory for storing intermediate/\+output images (i.\+e. working memory during processing), use one \href{https://github.com/dusty-nv/jetson-utils/tree/master/cuda/cudaMappedMemory.h}{\texttt{{\ttfamily cuda\+Alloc\+Mapped()}}} from C++ or the \href{https://rawgit.com/dusty-nv/jetson-inference/master/docs/html/python/jetson.utils.html\#cudaImage}{\texttt{{\ttfamily cuda\+Image}}} object from Python. Note that the \href{aux-streaming\#source-code}{\texttt{{\ttfamily video\+Source}}} input streams automatically allocate their own GPU memory, and return to you the latest image, so you needn\textquotesingle{}t allocate your own memory for those. ~\newline


Memory allocated by \href{https://github.com/dusty-nv/jetson-utils/tree/master/cuda/cudaMappedMemory.h}{\texttt{{\ttfamily cuda\+Alloc\+Mapped()}}} resides in a shared CPU/\+\+GPU memory space, so it is accessible from both the CPU and GPU without needing to perform a memory copy between them (hence it is also referred to as Zero\+Copy memory). ~\newline


Synchronization is required however -\/ so if you want to access an image from the CPU after GPU processing has occurred, call \href{https://docs.nvidia.com/cuda/cuda-runtime-api/group__CUDART__DEVICE.html\#group__CUDART__DEVICE_1g10e20b05a95f638a4071a655503df25d}{\texttt{{\ttfamily cuda\+Device\+Synchronize()}}} first. To free the memory in C++, use the \href{https://docs.nvidia.com/cuda/cuda-runtime-api/group__CUDART__MEMORY.html\#group__CUDART__MEMORY_1g71c078689c17627566b2a91989184969}{\texttt{{\ttfamily cuda\+Free\+Host()}}} function. In Python, the memory will automatically be released by the garbage collector, but you can free it explicitly with the {\ttfamily del} operator. ~\newline

So you can do things like {\ttfamily img.\+width} and {\ttfamily img.\+height} to access properties about the image.\hypertarget{md_jetson-inference_2docs_2aux-image_autotoc_md67}{}\doxysubsubsection{\texorpdfstring{Array Interfaces}{Array Interfaces}}\label{md_jetson-inference_2docs_2aux-image_autotoc_md67}
For zero-\/copy interoperability with other libraries, there exist several ways to access the {\ttfamily cuda\+Image} memory from Python:\+


\begin{DoxyItemize}
\item Indexing images directly from Python
\item Numpy \`{}\+\_\+\+\_\+array\+\_\+\+\_\+\`{} interface, \`{}cuda\+To\+Numpy()\`{}, \`{}cuda\+From\+Numpy()\`{}
\item Numba \`{}\+\_\+\+\_\+cuda\+\_\+array\+\_\+interface\+\_\+\+\_\+\`{} (Py\+Torch, Cu\+Py, Py\+CUDA, VPI, ect)
\item Sharing the Memory Pointer
\end{DoxyItemize}

These are implemented so that the underlying memory is mapped and shared with the other libraries as to avoid memory copies.\hypertarget{md_jetson-inference_2docs_2aux-image_autotoc_md68}{}\doxysubsubsection{\texorpdfstring{Accessing Image Data in Python}{Accessing Image Data in Python}}\label{md_jetson-inference_2docs_2aux-image_autotoc_md68}

\begin{quote}
{\bfseries{note:\+}} the Python subscripting index operator is only available if the cuda\+Image was allocated with {\ttfamily mapped=True} (which is the default). Otherwise, the data is not accessible from the CPU, and an exception will be thrown. \end{quote}


\doxylink{requirements_The}{The} indexing tuple used to access an image may take the following forms:\+


\begin{DoxyItemize}
\item {\ttfamily img\+[y,x]\+} -\/ note the ordering of the {\ttfamily (y,x)} tuple, same as numpy
\item {\ttfamily img\+[y,x,channel]\+} -\/ only access a particular channel (i.\+e. 0 for red, 1 for green, 2 for blue, 3 for alpha)
\item {\ttfamily img\+[y\texorpdfstring{$\ast$}{*}img.width+x]\+} -\/ flat 1D index, access all channels in that pixel
\end{DoxyItemize}

Although image subscripting is supported, individually accessing each pixel of a large image isn\textquotesingle{}t recommended to do from Python, as it will significantly slow down the application. Assuming that a GPU implementation isn\textquotesingle{}t available, a better alternative is to use Numpy.\hypertarget{md_jetson-inference_2docs_2aux-image_autotoc_md69}{}\doxysubsubsection{\texorpdfstring{Accessing as a Numpy Array}{Accessing as a Numpy Array}}\label{md_jetson-inference_2docs_2aux-image_autotoc_md69}
cuda\+Image supports the Numpy \href{https://numpy.org/doc/stable/reference/arrays.interface.html}{\texttt{{\ttfamily \+\_\+\+\_\+array\+\_\+\+\_\+}}} interface protocol, so it can be used in many Numpy functions as if it were a Numpy array without needing to copy it back and forth. See \href{https://github.com/dusty-nv/jetson-utils/blob/master/python/examples/cuda-to-numpy.py}{\texttt{{\ttfamily cuda-\/to-\/numpy.\+py}}} for a simple example:\+

\DoxyCodeLine{}
\DoxyCodeLine{print(np.add(cuda\_img,\ array))}

\end{DoxyCode}


\begin{quote}
{\bfseries{note:\+}} Numpy runs on the CPU, so the cuda\+Image should have been allocated with {\ttfamily mapped=True} (the default) so that it\textquotesingle{}s memory is accessible from both the CPU and GPU. Any changes to it from Numpy will be reflected in the underlying cuda\+Image\textquotesingle{}s memory. ~\newline
 \end{quote}


You\textquotesingle{}ll need to use the standalone \href{https://numpy.org/doc/stable/reference/routines.html}{\texttt{numpy routines}} as opposed to the \href{https://numpy.org/doc/stable/reference/generated/numpy.ndarray.html}{\texttt{ndarray class methods}} (e.\+g. use {\ttfamily numpy.\+mean(array) vs array.\+mean()}) because although cuda\+Image exports the {\ttfamily \+\_\+\+\_\+array\+\_\+\+\_\+} interface for accessing it\textquotesingle{}s memory, it doesn\textquotesingle{}t implement the class methods that Numpy does. To use all of those, see the \`{}cuda\+To\+Numpy()\`{} function below.\hypertarget{md_jetson-inference_2docs_2aux-image_autotoc_md70}{}\doxysubsubsubsection{\texorpdfstring{Converting to Numpy Arrays}{Converting to Numpy Arrays}}\label{md_jetson-inference_2docs_2aux-image_autotoc_md70}
You can explicitly obtain a \href{https://numpy.org/doc/stable/reference/generated/numpy.ndarray.html}{\texttt{{\ttfamily numpy.\+ndarray}}} object that\textquotesingle{}s mapped to a {\ttfamily cuda\+Image} by calling {\ttfamily cuda\+To\+Numpy()} on it:\+ ~\newline

As before, the underlying memory isn\textquotesingle{}t copied and Numpy will access it directly -\/ so if you change the data in-\/place through Numpy, it will be changed in the underlying {\ttfamily cuda\+Image} as well. For an example of using {\ttfamily cuda\+To\+Numpy()}, see the \href{https://github.com/dusty-nv/jetson-utils/blob/master/python/examples/cuda-to-numpy.py}{\texttt{{\ttfamily cuda-\/to-\/numpy.\+py}}} sample from jetson-\/utils.

{\ttfamily cuda\+Image} also supports Numba\textquotesingle{}s \href{https://numba.readthedocs.io/en/stable/cuda/cuda_array_interface.html}{\texttt{{\ttfamily \+\_\+\+\_\+cuda\+\_\+array\+\_\+interface\+\_\+\+\_\+}}}, which provides zero-\/copy interoperability among libraries that use GPU memory (including Py\+Torch, Cu\+Py, Py\+CUDA, VPI, and others \href{https://numba.readthedocs.io/en/stable/cuda/cuda_array_interface.html\#interoperability}{\texttt{listed here}}). Similar to how {\ttfamily cuda\+Image} implements the Numpy \`{}\+\_\+\+\_\+array\+\_\+\+\_\+\`{} interface, the {\ttfamily \+\_\+\+\_\+cuda\+\_\+array\+\_\+interface\+\_\+\+\_\+} is transparent to use for passing cuda\+Image\textquotesingle{}s into Numba/\+\+Py\+Torch/\+\+Cu\+Py/\+\+Py\+CUDA functions, and the memory is shared with these libraries so there aren\textquotesingle{}t memory copies or extra overhead.

For an example of this, see the code from \href{https://github.com/dusty-nv/jetson-utils/blob/master/python/examples/cuda-array-interface.py}{\texttt{{\ttfamily cuda-\/array-\/interface.\+py}}}:\+

Another example can be found in \href{https://github.com/dusty-nv/jetson-utils/blob/master/python/examples/cuda-to-pytorch.py}{\texttt{{\ttfamily cuda-\/to-\/pytorch.\+py}}} for sharing memory with Py\+Torch tensors:\+
