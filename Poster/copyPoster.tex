					\block{Problem Description}
					{
					Efficient vehicle handling at high-throughput gates (airport drop-off / parking) needs fast and reliable identification without manual checks. This project implements an \textbf{edge-first} License Plate Detection System on the \textbf{NVIDIA Jetson Nano}, using a single USB camera and fully on-device inference to support gate-level decisions.\newline
					
						The system observes vehicles at the lane, detects the license plate in real time, and extracts the alphanumeric text to identify the vehicle session. Our design goal is low latency with continuous operation: the pipeline runs locally (no cloud dependency) and maintains persistent state so it can recover cleanly after restarts. In operation, the camera continuously captures frames; a YOLOv8 detector accelerated with TensorRT localizes the license plate; the detected region is cropped and preprocessed; EasyOCR runs on the CPU to read the characters; and the recognized string is validated using an allowlist and German-plate formatting constraints before being stored.\newline
						
						For each recognized plate, the system logs an \textbf{entry timestamp} and an \textbf{exit timestamp} in a local \textbf{SQLite} database. The parking duration (dwell time) is computed as the difference between the exit and entry timestamps, enabling a configurable billing/time-limit decision. Results are presented as a live annotated video stream and a local web dashboard that lists recent plates together with timestamps and the computed fee.
						
					}
				
				\block{Challenges}
				{
				Deploying license plate recognition on a low-power edge device introduces practical constraints beyond model accuracy:\newline
				
					\begin{itemize}
						\item \textbf{Compute + memory limits (edge hardware)}\\
						The Jetson Nano has 4\,GB unified memory shared between CPU and GPU. Running YOLO detection, OCR, video encoding, and a live dashboard concurrently can exhaust RAM and reduce throughput. We had to keep the pipeline lightweight (efficient preprocessing, minimal buffering) and balance GPU detection with CPU OCR.
						
						\item \textbf{Storage constraints (production module)}\\
						Internal storage is limited (16\,GB eMMC) and our deployment does \textbf{not} use an SD card. Runtime artefacts (TensorRT engine, database, logs) must be stored on an external SSD and referenced via stable mount paths. This also affects reliability: abrupt shutdowns or write interruptions can corrupt state if storage is not handled carefully.
						
						\item \textbf{Runtime robustness}\\
						A gate system must run continuously and recover from transient failures. Long runs can trigger memory pressure, swap thrashing, camera read dropouts, and thermal throttling that reduces performance. Stable operation requires resource monitoring, conservative thresholds, and a design that keeps the system responsive under load.
						
						\item \textbf{Legacy software ecosystem}\\
						JetPack 4.6 (Ubuntu 18.04 / Python 3.6) constrains library versions and compatibility. Many modern ML packages expect newer Python/CUDA stacks, so deployment choices must fit the Jetson environment. This increases integration effort and motivates using exported/optimized artefacts (e.g., TensorRT engines) instead of heavy training stacks on-device.
						
						\item \textbf{Real-world imaging conditions}\\
						Lighting changes, glare, shadows, dirty/occluded plates, motion blur, and oblique viewing angles reduce OCR quality even when detection is correct. Camera placement (height/angle/distance) and exposure settings strongly influence readability. In addition, plate-like textures (logos/reflections) can cause false detections if thresholds are not tuned.
						
						\item \textbf{OCR ambiguity + validation}\\
						OCR often produces noisy strings: missing separators, swapped characters, and look-alike symbols (O/0, B/8). German plates also have variable prefix lengths and formatting rules, so we need allowlists and format validation to accept plausible outputs only. Without validation, incorrect reads quickly accumulate in the database and reduce trust in the system.
						
						% \item \textbf{Timing in real time}\\
						% OCR is much more expensive than detection, and running it on every frame is not feasible. The runtime must select when to OCR (stability window + cooldown) so it reacts quickly to new vehicles but avoids repeated OCR on near-identical frames. This trade-off directly affects throughput, responsiveness, and the perceived ``instant'' behavior at the gate.
					\end{itemize}
				
					
				}
			}

			\block{Solutions}
			{
			
			We combined hardware-aware deployment, optimized inference, and practical validation logic to achieve real-time operation on the Jetson Nano:
			
			\begin{itemize}
				\item \textbf{Hardware-aware deployment}\\
					Store runtime artefacts (TensorRT engine + database) on an \textbf{external SSD} mounted at \texttt{/media/myssd}.
					Use swap on SSD to reduce out-of-memory failures during long runs.
				
				
				\item \textbf{Real-time detection on GPU}\\
				Deploy the detector as a \textbf{TensorRT} engine (\texttt{best.engine}) for accelerated inference.
				Use a lightweight model variant suitable for stable edge performance.
				
				
				\item \textbf{OCR + validation on CPU}\\
				Run \textbf{EasyOCR} in CPU mode and preprocess plate crops (resize + thresholding) for improved readability.
				Apply an allowlist and German-format validation (city-prefix handling + numeric suffix rules) to reduce false reads.
				
				
				\item \textbf{Persistence + local UI}\\
				Log visits to a local SQLite database with entry and exit timestamps for each recognized plate.
				Provide a local Flask dashboard (live stream + billing table) for on-site monitoring and reporting.
					
				\end{itemize}
			
			}
			
			\block{Results}
			{
			The system runs end-to-end on the Jetson Nano and produces actionable, real-time outputs suitable for continuous edge operation. In our deployment, the detector maintains stable real-time throughput of approximately 13--15 FPS using TensorRT acceleration. The complete pipeline captures camera frames, detects the license plate, extracts text using OCR, validates the result, and records the visit in a local database before presenting the output on a live dashboard.
			The dashboard provides operationally useful information by listing recognized plates together with timestamps and a computed dwell-time fee derived from stored visit data. Because processing and logging are performed locally on the Jetson, the system remains responsive without relying on external servers. The final deployment runs on a Jetson Nano (4\,GB) with an external SSD used to store runtime artefacts and persistent logs.
			}
